# SOUL 模块问题清单

记录日期：2026-04-15

## 说明

本文档用于记录当前在 `nanobot/soul/` 相关改造中已经确认的问题、影响范围、严重级别与建议处理方向。

当前判断标准：

- `CORE_ANCHOR` / `SOUL_METHOD` 负责方法论与边界
- `SOUL_PROFILE` 负责结构化慢状态真源
- `SOUL` 负责自然语言人格与关系画像承载
- `HEART` 负责热状态承载
- 具体关系判断、关系表达、人格表达，应尽量由 LLM 在方法论约束下推演
- 程序负责结构、校验、限幅、留痕、兜底，不负责直接做产品语义判断

---

- 已处理：

## 问题 1：`SOUL.md` 的关系画像被程序模板化生成

- 涉及文件：
  - `nanobot/soul/projection.py`
- 严重级别：P1
- 问题描述：
  - 当前 `build_relationship_portrait()` 使用固定阈值和固定句子生成关系画像，例如：
  - `trust >= 0.6`
  - `intimacy >= 0.4`
  - `attachment >= 0.4`
  - 并将不同阶段映射为固定文案
- 本质问题：
  - 这已经不是“程序做兜底”，而是“程序直接解释关系状态并决定表达内容”
  - 与“由 LLM 基于方法论边界推演关系表达”的架构要求冲突
- 影响：
  - `SOUL.md` 的关系表达会被程序写死
  - 后续很难实现真正基于上下文、性格、历史记忆的动态关系画像
- 建议处理：
  - 将 `projection.py` 从“程序模板投影”改成“LLM 方法论投影”
  - 程序只负责把 `SOUL_PROFILE + 方法论 + 现有 SOUL` 组织成输入
  - 最终 `SOUL.md` 的文案由 LLM 生成

---

## 问题 2：初始化 fallback 仍然直接强制关系阶段

- 涉及文件：
  - `nanobot/soul/bootstrap.py`
- 严重级别：P1
- 问题描述：
  - `build_initial_profile()` 当前仍然直接写入：
  - `relationship["stage"] = "还不认识"`
- 本质问题：
  - 虽然之前已经把强制值从“熟悉”改成了“还不认识”，但依然是程序直接决定阶段
  - 这仍然不符合“由方法论约束下的 LLM 推演初始关系阶段”的原则
- 影响：
  - 一旦进入 fallback，初始化阶段判断就失去方法论推演能力
  - 未来不同初始语境下无法自然区分“还不认识 / 熟悉”
- 建议处理：
  - fallback 只提供空壳默认结构，不直接强指定阶段
  - 或者将 fallback 的阶段来源改为“方法论最低合法阶段 + 审计标记”
  - 最终应让初始化阶段的判断由 LLM 推演，程序只做合法性校验

---

## 问题 3：`HEART` 初始化内容仍然写死具体情感表达

- 涉及文件：
  - `nanobot/soul/heart.py`
- 严重级别：P2
- 问题描述：
  - `HeartManager.initialize()` 当前仍写入固定热状态内容，例如：
  - `对用户有些好奇，也在慢慢感受彼此之间的距离`
  - `中`
  - `刚刚开始，还没有趋势`
  - `想了解用户`
- 本质问题：
  - 这些不是结构定义，而是具体情绪与关系感受
  - 属于热状态内容，不应长期由程序写死
- 影响：
  - `HEART` 初始化会带入程序预设语气
  - 后续很难区分哪些是真实推演结果，哪些只是程序预设
- 建议处理：
  - 保留 `HEART` 的章节结构
  - 具体初始热状态改为由初始化 LLM 或上游 seed 生成
  - 程序只保留极薄兜底，例如空白占位或最弱默认文本

---

## 问题 4：初始化治理规则仍然把产品决策写死在代码里

- 涉及文件：
  - `nanobot/soul/init_adjudicator.py`
  - `nanobot/soul/init_inference.py`
- 严重级别：P2
- 问题描述：
  - 当前 init 裁决与 schema 中仍然写死：
  - 初始化允许阶段仅为 `还不认识 / 熟悉`
  - `relationship.boundary >= 0.5`
  - `companionship.boundary_expression >= 0.5`
- 本质问题：
  - 这些是治理规则，但目前仍散落在程序代码里
  - 如果未来要迭代方法论，就需要同步改代码
- 影响：
  - 方法论规则和程序耦合过紧
  - 不利于未来通过配置 / 方法论文档统一治理
- 建议处理：
  - 将这类规则逐步下沉到方法论配置层
  - 程序只读取规则并执行校验

---

## 问题 5：初始化链路会覆盖 LLM 生成的 `SOUL.md`

- 涉及文件：
  - `nanobot/soul/bootstrap.py`
- 严重级别：P1
- 问题描述：
  - 当前 `bootstrap_workspace()` 先写入一次 `SOUL.md`
  - 随后又调用 `sync_soul_from_profile(workspace)` 再次覆盖 `SOUL.md`
- 本质问题：
  - 初始化链路中同时存在两条 `SOUL` 生成路径
  - 后写入的一条会直接覆盖前面的结果
- 影响：
  - 即使 init 阶段的 LLM 生成了较好的 `SOUL.md`，最终也可能被程序模板化投影覆盖
  - 初始化结果语义不稳定，且不易察觉
- 建议处理：
  - 初始化时应明确只有一条 `SOUL` 生成链
  - 如果采用 LLM 投影，则 bootstrap 不应再先写一版静态 `SOUL`
  - 如果保留 seed 文案，也必须明确“谁是最后写入者”

---

## 问题 6：方法论真源已经发生分裂与漂移

- 涉及文件：
  - `nanobot/soul/methodology.py`
  - `nanobot/templates/SOUL_METHOD.md`
  - `nanobot/soul/bootstrap.py`
- 严重级别：P1
- 问题描述：
  - 关系阶段定义目前出现在多个地方：
  - `methodology.py`
  - `SOUL_METHOD.md`
  - `bootstrap.py` 里的 `build_soul_method_markdown()`
  - 并且它们已经不一致，例如代码里已有 `还不认识`，模板文档仍未同步
- 本质问题：
  - 方法论来源不唯一
  - 已经出现定义漂移
- 影响：
  - 初始化、裁决、文档、演化链使用的关系阶段定义可能不一致
  - 后续会导致“程序允许 / 文档不允许”或“文档存在 / 程序未知”的问题
- 建议处理：
  - 建立单一方法论真源
  - 其他位置全部从真源派生
  - 禁止再手写第二份阶段定义

---

## 问题 7：人格演化结果会过度覆盖 `SOUL.md` 性格画像

- 涉及文件：
  - `nanobot/soul/evolution.py`
  - `nanobot/soul/projection.py`
- 严重级别：P2
- 问题描述：
  - 当前人格演化后，会把 `manifestation` 直接作为 `personality_seed` 传给 `sync_soul_from_profile()`
  - 这意味着一次局部人格表现可能直接替换整段 `SOUL` 中的性格画像
- 本质问题：
  - 将“阶段性表现”错误提升成“长期画像文本”
  - 不符合慢变量稳定性要求
- 影响：
  - `SOUL.md` 会被单次演化结果频繁重写
  - 性格叙事容易漂移
- 建议处理：
  - 将 `manifestation` 作为周复盘或演化日志材料，而不是直接覆盖 `SOUL`
  - `SOUL` 的人格画像应由完整结构态投影生成，而不是由一次演化短句替换

---

## 问题 8：`SOUL_PROFILE` 虽然已停止直接注入，但其上游定位尚未完全收口

- 涉及文件：
  - `nanobot/soul/engine.py`
  - `nanobot/soul/review.py`
  - `nanobot/soul/evolution.py`
  - `nanobot/soul/projection.py`
- 严重级别：P2
- 问题描述：
  - 当前已经把 `SOUL_PROFILE` 直接注入上下文关闭
  - 但 `SOUL_PROFILE -> SOUL` 的投影机制仍未切换到 LLM 化
- 本质问题：
  - 当前“结构化真源”已经成立
  - 但“自然语言承载层”还未真正按架构要求完成
- 影响：
  - 整体链路还处在半收口状态
- 建议处理：
  - 下一阶段优先完成：
  - `SOUL_PROFILE -> SOUL` 的 LLM 投影
  - `HEART` 初始化去内容硬编码
  - 方法论单一真源整理

## 总结

当前最核心的问题不是单一文件里某几个字面量，而是：

- 方法论真源不唯一
- `SOUL_PROFILE -> SOUL` 仍然是程序模板投影，而非 LLM 方法论投影
- 初始化、投影、演化三条链存在互相覆盖与语义冲突

后续如果继续按当前方向局部修修补补，问题会不断转移位置，不会真正消失。

真正的收口方向应是：

1. 确立单一方法论真源
2. 让 `SOUL_PROFILE` 成为唯一结构化真源
3. 让 `SOUL` 成为 LLM 方法论投影后的自然语言画像
4. 让 `HEART` 只负责热状态
5. 程序只负责校验、限幅、留痕、兜底，不直接负责关系表达与人格表达
