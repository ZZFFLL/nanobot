# SOUL 状态架构整改实施计划

日期：2026-04-15

---

## 1. 目标

基于《SOUL 状态架构整改方案》，分阶段完成 `SOUL_METHOD / SOUL_PROFILE / SOUL / HEART` 的职责收口与链路重排，使系统满足：

- 方法论唯一真源
- `SOUL_PROFILE` 唯一结构化慢状态真源
- `SOUL` 唯一自然语言慢状态承载
- `HEART` 唯一热状态承载
- 关系阶段与关系画像由 LLM 在方法论边界下推演
- 程序只负责治理、校验、限幅、留痕

---

---

## 2. 实施原则

1. 优先修正结构错误，不先做表面补丁
2. 优先处理会导致链路覆盖和语义冲突的问题
3. 所有改动尽量限制在 `nanobot/soul` 与最小桥接层
4. 每一阶段都保证可测试、可验证

---

## 3. 分阶段实施

### Phase 1：方法论真源统一

#### 目标

统一关系阶段、关系维度、治理节奏的来源，消除定义漂移。

#### 需要处理的问题

- `methodology.py`、`SOUL_METHOD.md`、`bootstrap.py` 中的阶段定义不一致
- 文档与代码已经出现分裂

#### 实施内容

1. 确定唯一方法论真源结构
2. 统一关系阶段定义
3. 移除重复维护的阶段文本
4. 所有使用方改为从单一来源读取

#### 验收标准

- 关系阶段定义只维护一份
- 代码与文档不再漂移

---

### Phase 2：重建 `SOUL_PROFILE -> SOUL` 投影链

#### 目标

让 `SOUL.md` 不再由程序模板拼接，而是由 LLM 在方法论约束下投影生成。

#### 当前问题

- `projection.py` 使用固定阈值与固定文案
- `SOUL` 不是方法论投影，而是程序生成文案

#### 实施内容

1. 废弃程序关系画像模板
2. 新增 `SOUL` 投影推演协议
3. LLM 输入包括：
   - `SOUL_PROFILE`
   - `SOUL_METHOD`
   - 当前 `SOUL`
   - 必要的锚点边界
4. 程序输出校验：
   - 必须为纯自然语言
   - 不得混入结构化字段
   - 不得越界

#### 验收标准

- `SOUL` 不再由程序阈值和句子决定
- `SOUL` 文本由 LLM 投影生成

---

### Phase 3：初始化链闭环重建

#### 目标

让初始化阶段也遵循统一架构，不再出现多条写入链互相覆盖。

#### 当前问题

- 先写一版 `SOUL`，再被投影链覆盖
- fallback 仍然直接决定阶段
- `HEART` 初始化内容过厚

#### 实施内容

1. 初始化链只保留一条 `SOUL` 生成路径
2. 初始化 LLM 输出：
   - `SOUL_PROFILE`
   - `SOUL`
   - `HEART`
3. 程序只做结构校验和边界校验
4. fallback 只做最弱兜底，不直接替代推演

#### 验收标准

- 初始化后 `SOUL / SOUL_PROFILE / HEART` 语义一致
- 不再存在“先写后覆盖”问题

---

### Phase 4：周复盘链与演化链对齐

#### 目标

让慢变量演化链完全统一：先改结构，再改表达。

#### 当前问题

- 周复盘已经开始改 `SOUL_PROFILE`
- 但 `SOUL` 投影链不正确
- 人格演化结果会过度替换 `SOUL` 文案

#### 实施内容

1. 周复盘只更新 `SOUL_PROFILE`
2. 更新后触发 `SOUL` 重新投影
3. 演化 manifestation 改为日志材料，不直接覆盖 `SOUL`
4. `SOUL` 由完整结构态重新生成

#### 验收标准

- 周复盘、演化、`SOUL` 投影三条链语义统一
- `SOUL` 不再被局部演化结果粗暴覆盖

---

### Phase 5：收口 `HEART` 初始化与治理参数外置

#### 目标

把 `HEART` 从“内容硬编码”收紧到“结构固定 + 极薄兜底”。

#### 当前问题

- `HEART` 初始化热状态内容仍然写死
- init 裁决中的边界阈值仍在代码内

#### 实施内容

1. `HEART` 初始化仅保留章节结构
2. 具体内容由初始化 seed 或 LLM 生成
3. 将 init 阶段集合、边界阈值逐步转为方法论配置

#### 验收标准

- `HEART` 不再承载程序写死的情绪文案
- 治理阈值不再散落在实现代码中

---

## 4. 推荐执行顺序

### 第一阶段

- Phase 1：方法论真源统一
- Phase 2：重建 `SOUL_PROFILE -> SOUL` 投影链

### 第二阶段

- Phase 3：初始化链闭环重建

### 第三阶段

- Phase 4：周复盘链与演化链对齐
- Phase 5：收口 `HEART` 初始化与治理参数外置

---

## 5. 每阶段验证要求

### 通用验证要求

每阶段结束后至少验证：

1. 相关测试通过
2. 目标文件职责清晰
3. 不存在互相覆盖
4. 不引入新的程序表达层硬编码

### 重点验证点

#### 初始化验证

- `SOUL`
- `SOUL_PROFILE`
- `HEART`

三者初始结果必须一致。

#### 周复盘验证

- 先改 `SOUL_PROFILE`
- 再更新 `SOUL`
- `HEART` 不直接决定阶段

#### 对话验证

- 上下文中只使用 `SOUL + HEART + CORE_ANCHOR + SOUL_METHOD`
- 不直接把 `SOUL_PROFILE` 作为慢状态摘要注入

---

## 6. 风险提示

### 风险 1：LLM 投影不稳定

#### 应对

- 加入结构与边界校验
- 必要时记录 trace
- 提供 fallback 但不替代方法论判断

### 风险 2：阶段与文案不同步

#### 应对

- 强制走“先改 profile，再投影 SOUL”流程

### 风险 3：方法论文档再次漂移

#### 应对

- 禁止多处手写阶段定义
- 建立单一来源与派生机制

---

## 7. 当前实施优先级结论

如果只能先做一件事，优先做：

## Phase 2：重建 `SOUL_PROFILE -> SOUL` 投影链

原因：

- 这是目前最核心的架构偏差
- 也是后续初始化、复盘、演化链全部会依赖的基础
- 不先修它，其它链路都还会继续带着错误方向运行

---

## 8. 实施结论

本计划采用：

- 先统一真源
- 再修投影链
- 再修初始化链
- 再统一复盘与演化链
- 最后清理残余硬编码

该顺序可以在不破坏现有仓库整体稳定性的前提下，逐步把 `SOUL` 状态架构收口到正确方向。
